# DeepFlow 0.1.0 开发强制规范

> 本文件为 DeepFlow 0.1.0 (V4.0) 开发行为"宪法"，每次修改代码前必须先读。

---

## 一、契约笼子强制流程（V1.0 核心机制）

### 1.1 三层强制机制

```
┌─────────────────────────────────────┐
│  第一层：Phase 门禁（开发流程强制）   │
│  不通过验证 → 不得进入下一 Phase      │
├─────────────────────────────────────┤
│  第二层：AI 提示词约束（行为强制）    │
│  任务描述包含强制约束 → AI 必须执行    │
├─────────────────────────────────────┤
│  第三层：自动化检查（CI 强制）        │
│  每次提交/Phase 完成自动运行验证       │
└─────────────────────────────────────┘
```

### 1.2 Phase 门禁流程

```
开发模块 → 创建契约 → 实现代码 → 运行验证 → 判断结果
                                        ↓
                              全部通过？→ 是：写入 PROGRESS → 下一 Phase
                                        → 否：返工 → 重新验证
```

**关键规则**：
- 验证失败 = 失败，不是"基本完成"
- 不得声称完成，必须报告验证结果
- 主 Agent 必须复核验证结果

### 1.3 AI 开发任务模板

```markdown
# Phase [N] 开发任务：[模块名]

## 任务
开发 [模块名] 模块，实现 [功能描述]

## 🚨 强制约束（必须遵守，不得跳过）

### 1. 契约先行
- 必须先读 cage/templates/module_contract.yaml 了解契约格式
- 必须先创建 cage/[module].yaml 契约文件
- 契约必须包含：interface + behavior + boundaries
- 契约文件大小：simple ≤2K / medium ≤5K / complex ≤10K

### 2. 契约约束下开发
- 代码必须实现契约中定义的所有接口
- 代码行为必须符合契约中的 success/failure 定义
- 代码必须处理契约中定义的所有边界条件

### 3. 验证闭环
- 实现后必须运行：python cage/validate.py [module]
- 必须运行：python cage/check_standards.py [module].py
- 必须运行：python -m pytest tests/contract/test_[module].py
- 验证失败必须返工，不得声称完成

### 4. 报告规范
- 完成报告必须包含验证结果（输出文本）
- 不得说"基本完成"，只有"通过"或"失败"
- 验证失败 = 失败，不是"基本完成"

## 验收标准
□ cage/[module].yaml 存在且通过 validate.py
□ [module].py 通过 check_standards.py（P0=0）
□ tests/contract/test_[module].py 全部通过
□ 无 bare except
□ 所有公开方法有类型注解和 docstring
```

## 二、编码规范

### 2.1 P0 违规（zero tolerance）

| 规则 | 说明 |
|:---|:---|
| bare except | 禁止 `except:` 或 `except Exception:` |
| 未使用导入 | 禁止导入但未使用的模块 |
| 语法错误 | 代码必须能正常解析 |

### 2.2 P1 违规（质量门禁）

| 规则 | 说明 |
|:---|:---|
| 类型注解缺失 | 所有公开方法必须有返回值和参数类型注解 |
| Docstring 缺失 | 所有公开方法必须有 docstring |
| 文件超长 | 单文件 ≤ 500 行 |

### 2.3 P2 违规（优化建议）

| 规则 | 说明 |
|:---|:---|
| 重复代码 | DRY 原则 |
| 日志格式 | 使用 kwargs，禁止 f-string 日志 |

## 三、开发前检查清单

```bash
□ 1. 读取本文件（30秒）
□ 2. 创建 cage/[module].yaml 契约文件
□ 3. 确认模块边界（依赖/被依赖）
□ 4. 实现代码（在契约约束下）
□ 5. 运行验证：
     ├── python cage/validate.py [module]
     ├── python cage/check_standards.py [module].py
     └── python -m pytest tests/contract/test_[module].py
□ 6. 全部通过后才算完成
□ 7. 写入 PROGRESS 报告
```

## 四、验证失败处理流程

```
验证失败
  ↓
分析失败类型（契约/边界/规范）
  ↓
定位问题代码行
  ↓
修复
  ↓
重新运行直到通过
  ↓
提交（带验证报告）
```

**关键规则**：
- 验证失败不得进入下一阶段
- 修复后必须重新运行所有验证
- 验证结果必须写入 PROGRESS 报告

## 五、快速参考

### 常见陷阱

| 陷阱 | 后果 | 正确做法 |
|:---|:---|:---|
| `_normalize_threshold(x) * 100` | 8500（重复缩放） | 统一 0-100 尺度 |
| `except Exception:` | P0 违规 | 使用具体异常类型 |
| `logger.info(f"...")` | 日志格式违规 | 使用 kwargs |
| 跳过契约直接写代码 | 架构违反 | 先写契约再写代码 |
| 验证失败声称完成 | 严重违规 | 验证失败 = 失败 |
| **主 Agent 手动拼 task 参数** | **Orchestrator 执行简化版，管线不完整** | **通过 orchestrator_agent.py 启动，加载完整 Prompt** |

### 决策树

```
不确定？→ 查 V1_BLUEPRINT.md
架构问题？→ 停止，先确认设计
测试失败？→ 不进入下一模块
时间不够？→ 只完成一个模块，不要半成品
验证失败？→ 返工，不得声称完成
```
